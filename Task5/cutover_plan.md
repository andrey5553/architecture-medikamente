# Стратегия миграции

### Выбранная стратегия: Parallel run

| Критерий                       | Обоснование                                                                          |
| ------------------------------ | ------------------------------------------------------------------------------------ |
| Минимизация простоя            | Система работает 24/7 — нельзя останавливать приём пациентов                         |
| Возможность тестирования в бою | Новая система тестируется параллельно с текущей — снижает риски                      |
| Валидация данных               | Можно сравнивать данные в старой и новой системе (diff-анализ)                       |
| Гибкость отката                | Если новая система падает — продолжаем работать со старой                            |
| Поддержка бизнес-процессов     | Администраторы и врачи учатся на "дубле", не рискуя ошибиться с реальными пациентами |

> **Почему не другие?**  
> - **High-level design** — требует полной остановки, недопустимо.  
> - **Branch by abstraction** — слишком медленный для критичных мер (бэкапы, шифрование).

# Cutover-план

  Этап 1: Подготовка окружения
  Описание действия: Развернуть "теневую" копию системы для параллельного запуска
  Ключевые задачи: Настроить дубль PostgreSQL, MinIO, Keycloak<br>, Grafana - Настроить feature флаги (будут позволять запуск частей системы после готовности или отключать в случае проблем) 
  для включения параллельной записи<br> , кстати можно развернуть тот FeatBeat (на беке после достаточно просто настроить его работу с фичами)
  Ответственные: DevOps, Архитектор, Backend Senior Developer
  Время/Сроки: 1 месяц
  Риски и меры по снижению: Риск: Рассинхрон данных<br>Мера: Автоматический diff-контроль каждые 15 мин

  Этап 2: Двойная запись (dual-write)
  Описание действия: Все новые данные пишутся в обе системы: старую и новую
  Ключевые задачи: Настроить BFF: дублирование запросов<br>- Включить логирование в обеих системах<br>- Обеспечить идемпотентность операций
  Ответственные: Backend Senior Developer
  Время/Сроки: 1-2 месяц
  Риски и меры по снижению: Риск: Ошибки дублирования<br>Мера: Валидация по ключам (patient_id, appointment_id) + alerting + logging (пользуем grafana + opentelemetry)

  Этап 3: Параллельная работа
  Описание действия: Персонал работает в старой системе, новая — в режиме мониторинга
  Ключевые задачи: Обучить администраторов и врачей<br>- Собрать фидбэк по UX<br>- Проверить производительность и безопасность
  Ответственные: Системный/Бизнес аналитик
  Время/Сроки: 2-4 месяц
  Риски и меры по снижению: Риск: Сопротивление персонала<br>Мера: Поощрение за использование новой системы

  Этап 4:  Постепенное переключение
  Описание действия: Перевод функциональности по модулям
  Ключевые задачи: Перевести: запись пациентов → анализы → отчётность<br>- Отключить старые модули по одному<br>- Обновить интеграции (1C, лаборатория)
  Ответственные: PM, Team Lead, Backend Developer, DevOps
  Время/Сроки: 5-7 месяц
  Риски и меры по снижению: Риск: Ошибки в интеграциях<br>Мера: Canary-релиз + мониторинг ошибок

  Этап 5: Отключение старой системы
  Описание действия: Финальная валидация и отключение legacy
  Ключевые задачи: Провести финальный diff<br>- Заблокировать доступ к старой системе<br>- Перевести все нагрузки на новую
  Ответственные: PM, Team Lead, Backend Developer, DevOps
  Время/Сроки: 2 недели
  Риски и меры по снижению: Риск: Остались "живые" ссылки на старую систему<br>Мера: Редирект 301 + уведомление пользователей

  Этап 6:  Поддержка и оптимизация
  Описание действия: Мониторинг, обучение, обратная связь
  Ключевые задачи:  Собрать KPI<br>- Провести опросы<br>- Устранить баги и узкие места
  Ответственные: Системный/Бизнес аналитик, Техподдержка
  Время/Сроки: 6 месяцев
  Риски и меры по снижению: Риск: Снижение производительности<br>Мера: Профилирование + оптимизация SQL, кэширование
  

## Контекст: ключевые направления трансформации

Этот cutover-план реализует следующие **приоритетные меры**, выявленные ранее:

### Высокий приоритет
- Внедрение шифрования (Vault) (хранение секретов)
- Настройка бэкапов и восстановления
- Запуск аудита действий с данными
- Внедрение DLP-системы
- Утверждение политики удаления ПДн

### Средний приоритет
- Переход на Kubernetes
- Замена Excel на PostgreSQL
- Настройка CI/CD
- Введение классификации данных

### Низкий приоритет
- Автоматизация записи пациентов
- Обучение персонала
- Сбор обратной связи

> Все эти меры **встраиваются в cutover-план** на соответствующих этапах.

## Управление рисками производительности

| Мера                             | Риск производительности           | Мера по снижению                                   |
| -------------------------------- | --------------------------------- | -------------------------------------------------- |
| Шифрование at rest (Vault + SSE) | +15–30% latency при чтении файлов | Кэширование в Redis; шифрование только метаданных  |
| DLP-сканирование email           | Задержка доставки на 1–5 сек      | Асинхронная обработка; исключение внутренних писем |
| Аудит всех действий              | Рост нагрузки на БД на 20–40%     | Асинхронная запись в ClickHouse через Kafka        |
| API Gateway + фильтрация PII     | Доп. задержка 5–10 мс             | Кэширование ответов; edge-размещение               |
| CI/CD + security-сканы           | Удлинение pipeline на 3–10 мин    | Incremental scan; кэширование зависимостей         |

## Обратная совместимость и миграция

- **Двойная запись (dual-write)** — включается на этапе 2, остаётся до отключения legacy.
- **Feature флаги** — управляют включением шифрования, DLP, аудита.
- **Blue/Green деплой** — используется для ключевых сервисов (Keycloak, BFF).
- **Мониторинг**: Prometheus + Grafana — метрики: latency, error rate, throughput.

| Среднее время записи пациента         | < 2 мин  | Анализ логов портала |

| Удовлетворённость персонала           | > 80%    | Опросы               |
